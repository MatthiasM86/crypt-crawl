## Project status

This repository is a scaffolded but currently empty Godot 4.x project. The folder
structure, `project.godot`, `.gitignore`, and a `GameManager` autoload stub exist,
but **no gameplay code has been written yet** — no player movement, no combat, no
enemy AI, no level generation, no scenes at all. The next implementation milestone
is **Phase 4** in `docs/plan.md` (player movement via click-to-move, then attack +
hit feedback, then enemy AI, then ranged enemies/projectiles, then the level
generator, then lighting/fog-of-war).

Godot 4.x is not yet installed in this environment. The project files were
hand-authored (not generated by the editor) and have never been opened or run —
treat the file formats as unverified until the first real editor open.

## Opening the project

Once Godot 4.x is installed:

- **Editor GUI:** open Godot's Project Manager and import `project.godot` from
  this directory.
- **From a terminal (repo root):**
  ```
  godot -e --path .
  ```
  (`-e` / `--editor` opens the editor instead of running the game.)
- **macOS, if `godot` isn't on PATH:** run the binary inside the app bundle directly:
  ```
  /Applications/Godot.app/Contents/MacOS/Godot -e --path .
  ```
- The first open generates a local `.godot/` cache folder (import data, editor
  state). It's git-ignored and machine-specific — never commit it.
- There is no `run/main_scene` yet (no scenes exist), so pressing Run (F5) will
  prompt for a main scene — expected until Phase 4 produces one.

## Project structure

- `project.godot` — Godot 4.x project file (`config_version=5`, Forward+ renderer,
  desktop-focused per the plan).
- `autoloads/game_manager.gd` — the `GameManager` singleton, registered in
  `project.godot`'s `[autoload]` section. Currently an empty `extends Node` stub;
  Phase 4 will give it HP/death/restart responsibilities.
- `scenes/player/`, `scenes/enemies/`, `scenes/projectiles/`, `scenes/levels/` —
  one folder per planned scene category (`Player`, `Enemy` base + variants,
  `Projectile`, `LevelGenerator`). Convention: a scene's script lives alongside
  its `.tscn` in the same folder (no separate top-level `scripts/` tree).
- `assets/sprites/` — sprite/image assets. Empty for now; the PoC plan calls for
  placeholder colored-rectangle graphics before any real art.
- `docs/plan.md` — the design plan (German).

## Autoload convention

New cross-scene global state should follow the same pattern as `GameManager`: a
plain script under `autoloads/`, registered as `Name="*res://autoloads/name.gd"`
in `project.godot`'s `[autoload]` section.

## What this project is

`docs/plan.md` (in German) is a proof-of-concept plan for a dark, top-down action-roguelike ("Diablo-style") built in **Godot 4.x**, targeting desktop only for the PoC. The single question the PoC is meant to answer: *does the core combat loop feel fun?*

Key decisions from the plan that should guide the initial implementation:

- **Perspective/controls:** Top-down, mouse-driven (left click = move/attack, right click = optional skill, space/shift = dash). Isometric perspective is explicitly deferred post-PoC due to asset cost.
- **Combat model:** Real-time hack'n'slash. Attacks have a cooldown (~0.5s) with a hitbox in front of the player. Enemy attacks must telegraph with a windup before landing — this is called out as the core of the game feel and worth disproportionate implementation effort.
- **Stats:** Deliberately minimal — HP, damage, move speed only.
- **Loop:** Permadeath + procedural levels (rooms & corridors); death restarts the run. Meta-progression (Hades-style currency/permanent upgrades) is explicitly out of scope for the PoC.
- **Planned scene/node structure:** `Player`, `Enemy` (base scene with inherited variants for melee/ranged), `Projectile`, `LevelGenerator`, `GameManager` (HP, death, restart).
- **Core Godot nodes to use:** `CharacterBody2D` + `move_and_slide()` for movement, `NavigationRegion2D`/`NavigationAgent2D` for click-to-move and enemy pathfinding, `Area2D` for hitboxes/projectiles, `AnimationPlayer`/`AnimatedSprite2D` for attack timing/windups, `PointLight2D` + `LightOccluder2D` for torchlight/fog-of-war.
- **Enemy AI:** Simple 3-state machine — `Idle` → `Chasing` (sees player) → `Attacking` (in range, with windup).
- **Build order (per the plan):** player movement first, then attack + hit feedback (hit-flash, knockback, screenshake — prioritized heavily), then enemy AI, then ranged enemies/projectiles, then the level generator (with per-level navmesh baking and a flood-fill reachability check), then lighting/fog-of-war and floor transitions.
- **PoC success criterion:** fighting 3–4 enemies at once should feel "crunchy" enough that repeated runs are appealing on their own. If combat feels mushy, the plan explicitly says to fix step 2 (hit feedback) before adding new features.
- **Explicitly out of scope for the PoC:** pixel art, meta-progression, biomes, bosses, item variety, skill trees, sound, balancing, monetization.

When implementing features, prefer the ordering and scope boundaries above — this is a feel-first prototype, not a full game.
